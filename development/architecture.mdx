CodePilot is a multi-model AI agent desktop client built on a layered architecture. Electron 40 provides the native desktop shell, Next.js 16 handles all frontend rendering and the internal REST API, better-sqlite3 stores all data locally, and the Claude Agent SDK drives AI interactions. Understanding how these layers connect is the starting point for any contribution.

## Tech stack

| Layer | Technology |
|---|---|
| Desktop shell | Electron 40 |
| Frontend framework | Next.js 16 (App Router) + React 19 |
| Styling | Tailwind CSS 4 + Radix UI |
| Database | better-sqlite3 (WAL mode) |
| AI integration | Claude Agent SDK, `@ai-sdk/anthropic`, `@ai-sdk/google`, `@ai-sdk/openai` |
| IM integration | Telegram Bot API, Feishu SDK (`@larksuiteoapi/node-sdk`), Discord.js |
| Code highlighting | Shiki |
| Markdown rendering | react-markdown, streamdown, markdown-it |
| Packaging | electron-builder (DMG + NSIS + AppImage) |
| Testing | Playwright (E2E), tsx + node:test (unit) |

## Data flow

The main chat path flows from the user interface through the API layer into the Claude Agent SDK and back:

```
User input → MessageInput component
           → POST /api/chat
           → claude-client.ts (creates SDK conversation)
           → Claude Agent SDK SSE stream
           → stream-session-manager.ts manages the stream
           → useSSEStream hook subscribes
           → MessageList renders
           → db.ts persists to SQLite
```

The Bridge subsystem adds a parallel path for remote IM control:

```
Telegram / Feishu message
  → Adapter (long polling / WebSocket)
  → channel-router.ts routes to CodePilot session
  → conversation-engine.ts calls Claude SDK
  → SDK SSE response
  → delivery-layer.ts formats + chunks
  → Adapter sends back to IM
```

## Key modules

<CardGroup cols={2}>
  <Card title="claude-client.ts" icon="brain">
    Wraps the Claude Agent SDK. Exports `streamClaude()`, which is the single entry point for all AI interactions — both the desktop UI and the Bridge subsystem call this function.
  </Card>
  <Card title="stream-session-manager.ts" icon="activity">
    Manages SSE stream lifecycle: tracks active streams, handles pause/resume, and exposes the stream state to React hooks via `useSSEStream`.
  </Card>
  <Card title="conversation-registry.ts" icon="database">
    Global in-process registry of active Claude SDK sessions. Allows interrupting or rewinding a session by its ID without re-creating it.
  </Card>
  <Card title="db.ts" icon="hard-drive">
    Single file that owns the entire SQLite schema, all migrations, and every CRUD function. All database access goes through this module — no other file calls `better-sqlite3` directly.
  </Card>
</CardGroup>

## Adding a new feature

When adding a feature that spans multiple layers, you will typically need to touch the following files:

| Touch point | Path | Purpose |
|---|---|---|
| Type definitions | `src/types/index.ts` | Add interfaces / types |
| Database | `src/lib/db.ts` | Add table or column (with migration) |
| API route | `src/app/api/{feature}/route.ts` | New REST endpoint |
| Page | `src/app/{feature}/page.tsx` | New route page |
| Components | `src/components/{feature}/` | UI components |
| Hook | `src/hooks/use{Feature}.ts` | State management hook |
| i18n | `src/i18n/en.ts` + `zh.ts` | Bilingual translation keys |
| Bridge | `src/lib/bridge/adapters/` + `types.ts` | If IM integration is involved |

<Note>
Always add both English and Chinese translation keys together. The app ships with full `en` and `zh` locale support and missing keys produce visible gaps in the Chinese UI.
</Note>
